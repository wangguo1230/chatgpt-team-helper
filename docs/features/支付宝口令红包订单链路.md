# 支付宝口令红包订单链路（配置到使用）

最后更新：2026-03-17 17:29:20 CST

## 1. 功能目标

在现有订单管理能力上，新增“支付宝口令红包订单”闭环，满足以下业务场景：

1. 用户可在公开页（无需登录）提交邮箱 + 支付宝口令 + 备注。
2. 每次提交会生成（或补录更新）订单，并占用一条现有 `alipay_redpack` 渠道兑换码。
3. 创建/补录事件按汇总策略邮件通知管理员。
4. 管理员在后台可执行“快速邀请”“同步状态”“账号同步自查”。
5. 订单状态按业务链路闭环推进：`pending -> invited -> redeemed`。
6. 快速邀请会消耗该订单绑定的兑换码，并在订单中保留兑换码记录。

## 2. 业务规则（已落地）

1. 默认状态是 `pending`（待处理）。
2. 快速邀请成功且邮件/邀请已发出后，状态置为 `invited`（已邀请）。
3. 发现邮箱已成为账号成员后，状态置为 `redeemed`（已兑换）。
4. 邀请失败时保持 `pending`，并记录失败原因到“邀请结果”。
5. 支付宝口令全局唯一。
6. 同邮箱 + 不同支付宝口令，视为多笔订单。
7. 公开页字段：邮箱、支付宝口令、备注（可选）；不需要手机号。
8. 创建/补录都会给管理员邮箱发通知。
9. 管理端展示口令不脱敏（按需求）。
10. 当前公开页不启用风控（按需求）。
11. 公开创建不会新建兑换码，只会占用已有 `channel=alipay_redpack` 的可用兑换码。
12. 快速邀请成功后会将该兑换码置为已消耗，并写入订单兑换码消耗时间；退回订单会释放占用。

## 3. 状态机与动作映射

| 触发动作 | 条件 | 状态变化 | 备注 |
|---|---|---|---|
| 公开提交 `POST /api/alipay-redpack/orders` | 口令不存在 | `pending` | 新建订单并邮件通知管理员 |
| 公开补录 `POST /api/alipay-redpack/orders/supplement` | 口令不存在 | `pending` | 补录新建并邮件通知管理员 |
| 公开补录 `POST /api/alipay-redpack/orders/supplement` | 口令已存在且邮箱一致 | 状态不变 | 仅更新备注（如传入）并邮件通知管理员 |
| 公开提交/补录 | 创建或命中订单 | 状态不变 | 自动占用一条 `alipay_redpack` 兑换码（库存不足则失败） |
| 管理端快速邀请 | 邀请成功且目标邮箱在邀请队列 | `pending/invited -> invited` | 记录邀请账号与邀请结果 |
| 管理端快速邀请 | 邀请成功且目标邮箱已入组 | `pending/invited -> redeemed` | 记录兑换完成 |
| 管理端快速邀请 | 邀请失败 | `pending/invited -> pending` | 记录失败原因 |
| 管理端快速邀请 | 邀请请求发出成功 | 状态按上方规则推进 | 消耗订单绑定兑换码，并记录消耗时间 |
| 管理端同步状态 | 邮箱已入组 | `pending/invited -> redeemed` | 用于二次校准 |
| 管理端同步状态 | 邮箱仍在邀请队列 | `pending/invited -> invited` | 用于二次校准 |

## 4. 部署前配置（运维/管理员）

### 4.1 邮件告警配置（必须）

后端 `.env` 至少配置：

```env
SMTP_HOST=smtp.example.com
SMTP_PORT=465
SMTP_SECURE=true
SMTP_USER=your-email@example.com
SMTP_PASS=your-password
SMTP_FROM=noreply@example.com
ADMIN_ALERT_EMAIL=admin@example.com
```

说明：

1. `ADMIN_ALERT_EMAIL` 是订单创建/补录通知的接收邮箱。
2. 未配置 SMTP 或管理员邮箱时，系统会跳过邮件发送并打印日志。

### 4.2 启动与数据库初始化

1. 启动后端时会自动执行数据库初始化。
2. 会自动创建（或补齐）`alipay_redpack_orders` 表及索引。
3. 会自动注册后台菜单键 `alipay_redpack_orders`，路径 `/admin/alipay-redpack-orders`。

### 4.3 菜单与角色授权

若管理员账号无法看到页面，请在“角色管理/菜单管理”里确保角色具备菜单权限：

- `alipay_redpack_orders`

### 4.4 防刷与告警冷却配置（建议）

后端 `.env` 可选增加：

```env
ALIPAY_REDPACK_PUBLIC_RATE_LIMIT_WINDOW_SEC=60
ALIPAY_REDPACK_PUBLIC_RATE_LIMIT_MAX=30
ALIPAY_REDPACK_ADMIN_ALERT_SUMMARY_INTERVAL_SEC=600
ALIPAY_REDPACK_ADMIN_ALERT_SUMMARY_MAX_EVENTS=500
```

说明：

1. 公开提交/补录默认按 IP 每 60 秒最多 30 次请求，超限返回 `429`。
2. 管理员订单告警默认每 10 分钟汇总发送一次。
3. 汇总队列默认最多保留 500 条，超出后会丢弃最旧记录。

## 5. 前端入口与页面使用

### 5.1 公开页（用户侧）

- 路由：`/redeem/alipay-redpack`
- 库存展示：
  - 显示“当前可兑换库存”（`alipay_redpack` 渠道未兑换且未被占用的兑换码数量）。
  - 支持手工“刷新库存”。
- 字段：
  - 邮箱（必填）
  - 支付宝口令（必填）
  - 备注（可选）

按钮行为：

1. `提交订单`：创建新订单（口令必须未被占用）。
2. `补录`：
   - 若口令不存在：创建订单。
   - 若口令已存在且邮箱一致：更新该订单备注。
   - 若口令已存在但邮箱不同：拒绝（防止改绑订单）。
3. 创建/补录成功后，系统会占用一个现有兑换码，不会按口令自动建码。

### 5.2 后台页（管理员侧）

- 路由：`/admin/alipay-redpack-orders`
- 展示字段：
  - 邮箱
  - 支付宝口令（明文展示）
  - 兑换码记录（兑换码 ID、是否已消耗、消耗时间）
  - 状态
  - 邀请结果
  - 备注
  - 操作人
  - 时间（创建/更新）

操作按钮：

1. `快速邀请`
   - 调用快捷邀请能力，自动选择或指定账号。
   - 成功后按队列状态写入 `invited` 或 `redeemed`。
   - 同时消耗该订单绑定兑换码（失败会回滚本次消耗）。
2. `同步状态`
   - 手工触发邀请/成员队列重新检查，修正订单状态。
3. `账号同步自查`
   - 自动跳转到账号管理页并携带 `syncAccountId`。
4. `退回订单`
   - 退回不可用口令订单，释放该订单占用的兑换码库存。

## 6. 账号管理联动（同步自查）

从订单页点击“账号同步自查”后：

1. 跳转到 `/admin/accounts?syncAccountId=<账号ID>`。
2. 账号页加载完成后自动触发该账号“同步用户数/邀请数”。
3. 同步结束后自动清理 query，避免重复触发。
4. 若当前分页找不到该账号，会回退到按 ID 拉取账号后再执行同步。

## 8. 日志与排障

### 8.1 公开提交成功但没收到管理员邮件

排查顺序：

1. 检查 `.env` 是否配置 `ADMIN_ALERT_EMAIL`。
2. 检查 SMTP 参数是否可连通并可投递。
3. 查看后端日志中是否有 `[AdminAlert]` 相关告警。

### 8.2 快速邀请后订单未变更为已兑换

1. 先点“同步状态”再次校准。
2. 若仍未变化，点“账号同步自查”触发目标账号同步。
3. 检查邀请结果字段和账号管理页的邀请/成员列表。

### 8.3 提示“口令已被其他邮箱使用，不能改绑邮箱”

这是预期保护逻辑：

- 防止公开补录被滥用来篡改历史订单归属。
- 需要管理员介入时，请在后台直接处理该订单。

### 8.4 提示“提交过于频繁，请 N 秒后再试”

这是公开接口限流生效的正常表现：

1. 等待提示秒数后重试。
2. 如确有业务高峰，可适当调高 `ALIPAY_REDPACK_PUBLIC_RATE_LIMIT_MAX`。
3. 如需更快收到邮件，可缩短 `ALIPAY_REDPACK_ADMIN_ALERT_SUMMARY_INTERVAL_SEC`。

### 8.5 兑换码已创建但兑换码管理页未看到

排查顺序：

1. 在兑换码管理页清空搜索条件后刷新。
2. 检查该口令是否已被其他渠道兑换码占用（会导致订单创建直接报冲突）。
3. 重点检索 `channel=alipay_redpack` 或按口令全文搜索。

## 9. 回归测试建议

每次变更建议至少执行：

1. 后端测试：`npm run test --workspace=backend`
2. 前端类型检查：`npm run type-check --workspace=frontend`
3. 手工流程：
   - 打开公开页 -> 可看到“当前可兑换库存”
   - 公开提交一条新订单 -> 后台看到 `pending`
   - 管理端快速邀请 -> 订单变 `invited` 或 `redeemed`
   - 点“账号同步自查” -> 账号页自动触发同步
   - 在订单页执行“同步状态” -> 状态与实际队列一致

## 10. 安全与风险说明

1. 公开页当前无风控（按业务要求）。
2. 已加防改绑保护：补录不允许跨邮箱接管已有口令订单。
3. 依赖层面已完成非破坏性漏洞修复；剩余漏洞集中在 `node-telegram-bot-api -> request` 旧链路（第三方包历史依赖），不影响本订单链路主功能，但建议后续专项治理 Telegram Bot 依赖。
