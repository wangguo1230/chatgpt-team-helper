# 支付宝口令红包订单链路（配置到使用）

最后更新：2026-03-18 10:40:00 CST

## 1. 功能目标

在现有订单管理能力上，新增“支付宝口令红包订单”闭环，满足以下业务场景：

1. 用户可在公开页（无需登录）提交邮箱 + 支付宝口令 + 备注。
2. 每次提交会生成（或补录更新）订单，并占用一条现有 `alipay_redpack` 渠道兑换码。
3. 创建/补录事件按汇总策略邮件通知管理员。
4. 管理员在后台可执行“快速邀请”“同步状态”“账号同步自查”。
5. 订单状态按业务链路闭环推进：`pending -> invited -> redeemed`。
6. 快速邀请会消耗该订单绑定的兑换码，并在订单中保留兑换码记录。

## 2. 业务规则（已落地）

1. 默认状态是 `pending`（待处理）。
2. 快速邀请成功且邮件/邀请已发出后，状态置为 `invited`（已邀请）。
3. 发现邮箱已成为账号成员后，状态置为 `redeemed`（已兑换）。
4. 邀请失败时保持 `pending`，并记录失败原因到“邀请结果”。
5. 支付宝口令全局唯一。
6. 同邮箱 + 不同支付宝口令，视为多笔订单。
7. 公开下单页字段：邮箱、支付宝口令、备注（可选）；不需要手机号。
8. 创建/补录都会给管理员邮箱发通知。
9. 管理端展示口令不脱敏（按需求）。
10. 公开补录链路启用 `OTP + 会话票据` 鉴权，候选查询/提交补录均需票据。
11. 公开接口启用按 IP 限流，超限返回 `429 + Retry-After`。
12. 公开创建不会新建兑换码，只会占用已有 `channel=alipay_redpack` 的可用兑换码。
13. 快速邀请成功后会将该兑换码置为已消耗，并写入订单兑换码消耗时间；退回订单会释放占用。

## 3. 状态机与动作映射

| 触发动作 | 条件 | 状态变化 | 备注 |
|---|---|---|---|
| 公开提交 `POST /api/alipay-redpack/orders` | 口令不存在 | `pending` | 新建订单并邮件通知管理员 |
| 发送补录验证码 `POST /orders/supplement/auth/send-code` | OTP 开启 | 无 | 受发送冷却 + IP 限流控制 |
| 校验验证码 `POST /orders/supplement/auth/verify-code` | 验证码有效 | 无 | 签发短时补录票据 |
| 查询候选 `GET /orders/supplement/candidates` | 票据有效 | 无 | 返回该邮箱可补录订单候选 |
| 提交补录 `POST /orders/supplement` | 票据有效 + 订单可补录 | 不回退 `pending` | 执行补录并写入补录记录（成功/失败/人工介入） |
| 公开提交成功 | 创建新订单 | 状态不变 | 自动占用一条 `alipay_redpack` 兑换码（库存不足则失败） |
| 管理端快速邀请 | 邀请成功且目标邮箱在邀请队列 | `pending/invited -> invited` | 记录邀请账号与邀请结果 |
| 管理端快速邀请 | 邀请成功且目标邮箱已入组 | `pending/invited -> redeemed` | 记录兑换完成 |
| 管理端快速邀请 | 邀请失败 | `pending/invited -> pending` | 记录失败原因 |
| 管理端快速邀请 | 邀请请求发出成功 | 状态按上方规则推进 | 消耗订单绑定兑换码，并记录消耗时间 |
| 管理端同步状态 | 邮箱已入组 | `pending/invited -> redeemed` | 用于二次校准 |
| 管理端同步状态 | 邮箱仍在邀请队列 | `pending/invited -> invited` | 用于二次校准 |

## 4. 部署前配置（运维/管理员）

### 4.1 邮件告警配置（必须）

后端 `.env` 至少配置：

```env
SMTP_HOST=smtp.example.com
SMTP_PORT=465
SMTP_SECURE=true
SMTP_USER=your-email@example.com
SMTP_PASS=your-password
SMTP_FROM=noreply@example.com
ADMIN_ALERT_EMAIL=admin@example.com
```

说明：

1. `ADMIN_ALERT_EMAIL` 是订单创建/补录通知的接收邮箱。
2. 未配置 SMTP 或管理员邮箱时，系统会跳过邮件发送并打印日志。

### 4.2 启动与数据库初始化

1. 启动后端时会自动执行数据库初始化。
2. 会自动创建（或补齐）`alipay_redpack_orders` 表及索引。
3. 会自动注册后台菜单键：
   - `alipay_redpack_orders`（`/admin/alipay-redpack-orders`）
   - `alipay_redpack_supplements`（`/admin/alipay-redpack-supplements`）

### 4.3 菜单与角色授权

若管理员账号无法看到页面，请在“角色管理/菜单管理”里确保角色具备菜单权限：

- `alipay_redpack_orders`
- `alipay_redpack_supplements`

### 4.4 防刷与告警冷却配置（建议）

后端 `.env` 可选增加：

```env
ALIPAY_REDPACK_PUBLIC_RATE_LIMIT_WINDOW_SEC=60
ALIPAY_REDPACK_PUBLIC_RATE_LIMIT_MAX=30
ALIPAY_REDPACK_ADMIN_ALERT_SUMMARY_INTERVAL_SEC=600
ALIPAY_REDPACK_ADMIN_ALERT_SUMMARY_MAX_EVENTS=500
ALIPAY_REDPACK_SUPPLEMENT_REQUIRE_OTP=true
ALIPAY_REDPACK_SUPPLEMENT_OTP_TTL_SEC=300
ALIPAY_REDPACK_SUPPLEMENT_OTP_SEND_COOLDOWN_SEC=60
ALIPAY_REDPACK_SUPPLEMENT_TICKET_TTL_SEC=900
ALIPAY_REDPACK_SUPPLEMENT_TICKET_MAX_USES=8
ALIPAY_REDPACK_SUPPLEMENT_MAX_ATTEMPTS_PER_ORDER=3
ALIPAY_REDPACK_SUPPLEMENT_ATTEMPT_WINDOW_SEC=86400
ALIPAY_REDPACK_SUPPLEMENT_ORDER_LOCK_SEC=120
```

说明：

1. 公开提交/补录默认按 IP 每 60 秒最多 30 次请求，超限返回 `429`。
2. 管理员订单告警默认每 10 分钟汇总发送一次。
3. 汇总队列默认最多保留 500 条，超出后会丢弃最旧记录。
4. 公开补录默认必须先完成邮箱验证码验证（可通过 `ALIPAY_REDPACK_SUPPLEMENT_REQUIRE_OTP=false` 关闭）。
5. 补录票据默认 15 分钟有效、最多使用 8 次，适配“查询候选 + 提交补录”连续操作。
6. 同一订单默认 24 小时内最多补录 3 次，并带 120 秒执行锁防并发重复执行。

## 5. 前端入口与页面使用

### 5.1 公开页（用户侧）

- 下单页路由：`/redeem/alipay-redpack`
- 补录页路由：`/redeem/alipay-redpack/supplement`
- 库存展示：
  - 显示“当前可兑换库存”（`alipay_redpack` 渠道未兑换且未被占用的兑换码数量）。
  - 支持手工“刷新库存”。
- 下单字段：
  - 邮箱（必填）
  - 支付宝口令（必填）
  - 备注（可选）

公开补录流程：

1. `发送验证码`：向邮箱发送 6 位验证码（受发送冷却限制）。
2. `验证邮箱`：验证码通过后签发短时补录票据。
3. `查询订单`：携带票据查询该邮箱可补录候选订单。
4. `提交订单补录`：选择订单后发起补录，仅 `invited/redeemed` 且在质保期内的订单可自动处理。

### 5.2 后台页（管理员侧）

- 路由：
  - `/admin/alipay-redpack-orders`
  - `/admin/alipay-redpack-supplements`
- 展示字段：
  - 邮箱
  - 支付宝口令（明文展示）
  - 兑换码记录（兑换码 ID、是否已消耗、消耗时间）
  - 状态
  - 邀请结果
  - 备注
  - 操作人
  - 时间（创建/更新）

操作按钮：

1. `快速邀请`
   - 调用快捷邀请能力，自动选择或指定账号。
   - 成功后按队列状态写入 `invited` 或 `redeemed`。
   - 同时消耗该订单绑定兑换码（失败会回滚本次消耗）。
2. `同步状态`
   - 手工触发邀请/成员队列重新检查，修正订单状态。
3. `账号同步自查`
   - 自动跳转到账号管理页并携带 `syncAccountId`。
4. `退回订单`
   - 退回不可用口令订单，释放该订单占用的兑换码库存。

## 6. 账号管理联动（同步自查）

从订单页点击“账号同步自查”后：

1. 跳转到 `/admin/accounts?syncAccountId=<账号ID>`。
2. 账号页加载完成后自动触发该账号“同步用户数/邀请数”。
3. 同步结束后自动清理 query，避免重复触发。
4. 若当前分页找不到该账号，会回退到按 ID 拉取账号后再执行同步。

## 7. 关键接口清单

### 7.1 公开接口（无需登录）

1. `GET /api/alipay-redpack/stock`
2. `POST /api/alipay-redpack/orders`
3. `POST /api/alipay-redpack/orders/supplement/auth/send-code`
4. `POST /api/alipay-redpack/orders/supplement/auth/verify-code`
5. `GET /api/alipay-redpack/orders/supplement/candidates`
6. `POST /api/alipay-redpack/orders/supplement`

### 7.2 管理接口（需登录 + 菜单权限）

1. `GET /api/alipay-redpack/admin/orders`
2. `POST /api/alipay-redpack/admin/orders/:id/quick-invite`
3. `POST /api/alipay-redpack/admin/orders/:id/sync-status`
4. `PATCH /api/alipay-redpack/admin/orders/:id/note`
5. `GET /api/alipay-redpack/admin/supplements`
6. `POST /api/alipay-redpack/admin/supplements/:id/retry`
7. `PATCH /api/alipay-redpack/admin/supplements/:id/manual-close`

鉴权要求：

- `authenticateToken`
- `requireMenu('alipay_redpack_orders')` 或 `requireMenu('alipay_redpack_supplements')`

## 8. 日志与排障

### 8.1 公开提交成功但没收到管理员邮件

排查顺序：

1. 检查 `.env` 是否配置 `ADMIN_ALERT_EMAIL`。
2. 检查 SMTP 参数是否可连通并可投递。
3. 查看后端日志中是否有 `[AdminAlert]` 相关告警。

### 8.2 快速邀请后订单未变更为已兑换

1. 先点“同步状态”再次校准。
2. 若仍未变化，点“账号同步自查”触发目标账号同步。
3. 检查邀请结果字段和账号管理页的邀请/成员列表。

### 8.3 提示“口令已被其他邮箱使用，不能改绑邮箱”

这是预期保护逻辑：

- 防止公开补录被滥用来篡改历史订单归属。
- 需要管理员介入时，请在后台直接处理该订单。

### 8.4 提示“提交过于频繁，请 N 秒后再试”

这是公开接口限流生效的正常表现：

1. 等待提示秒数后重试。
2. 如确有业务高峰，可适当调高 `ALIPAY_REDPACK_PUBLIC_RATE_LIMIT_MAX`。
3. 如需更快收到邮件，可缩短 `ALIPAY_REDPACK_ADMIN_ALERT_SUMMARY_INTERVAL_SEC`。
4. 补录链路请确认已完成 OTP 验证并携带有效票据。

### 8.5 兑换码已创建但兑换码管理页未看到

排查顺序：

1. 在兑换码管理页清空搜索条件后刷新。
2. 检查该口令是否已被其他渠道兑换码占用（会导致订单创建直接报冲突）。
3. 重点检索 `channel=alipay_redpack` 或按口令全文搜索。

## 9. 回归测试建议

每次变更建议至少执行：

1. 后端测试：`npm run test --workspace=backend`
2. 前端类型检查：`npm run type-check --workspace=frontend`
3. 手工流程：
   - 打开公开页 -> 可看到“当前可兑换库存”
   - 公开提交一条新订单 -> 后台看到 `pending`
   - 管理端快速邀请 -> 订单变 `invited` 或 `redeemed`
   - 点“账号同步自查” -> 账号页自动触发同步
   - 在订单页执行“同步状态” -> 状态与实际队列一致

## 10. 安全与风险说明

1. 公开补录链路启用 OTP 鉴权、票据绑定（邮箱/IP/UA）与按 IP 限流。
2. 已加防改绑保护：补录不允许跨邮箱接管已有口令订单。
3. 补录执行具备订单级互斥锁与次数窗口限制，降低并发重复消耗风险。
4. 依赖层面已完成非破坏性漏洞修复；剩余漏洞集中在 `node-telegram-bot-api -> request` 旧链路（第三方包历史依赖），不影响本订单链路主功能，但建议后续专项治理 Telegram Bot 依赖。
