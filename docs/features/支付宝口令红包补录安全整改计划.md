# 支付宝口令红包补录安全整改计划（订单系统）

最后更新：2026-03-18 09:35:00 CST

## 1. 目标与范围

### 1.1 目标

围绕支付宝口令红包“订单补录”链路，完成以下目标：

1. 补录流程不可被未授权触发（防越权、防枚举、防刷）。
2. 订单状态机可追踪、可回放，不出现状态回退导致的重复处理。
3. 库存占用/消耗/释放强一致，避免重复消耗和库存泄漏。
4. 质保判定锚点固定，补录不会延长质保窗口。
5. 公开接口最小化暴露，保障敏感信息安全。

### 1.2 审计范围

1. 公开补录入口：`/api/alipay-redpack/orders/supplement/candidates`、`/api/alipay-redpack/orders/supplement`
2. 订单核心服务：`backend/src/services/alipay-redpack-orders.js`
3. 补录执行器：`executeAlipayRedpackOrderSupplement`
4. 后台补录管理：`/api/alipay-redpack/admin/supplements*`
5. 前端补录页与订单管理页：`AlipayRedpackSupplementView`、`AlipayRedpackOrdersView`

## 2. 风险分级结论

### 2.1 严重（必须先修）

1. 公开补录缺少强身份校验，仅依赖 `email + orderId`，存在未授权触发风险。
2. 同一订单可重复补录并重复消耗兑换码，存在库存被刷空风险。
3. 补录会重置质保时间起点，违反“补录不延长质保”的业务约束。

### 2.2 高风险（本轮必须修）

1. 候选查询接口未限流，且错误语义可用于存在性探测。
2. 邀请已成功但同步失败时状态回退 `pending`，引发重复处理语义混乱。
3. 退回订单未显式撤销外部邀请，可能出现“邀请仍可接受但库存已释放”。

### 2.3 中风险（P1）

1. 公开接口返回字段过多（口令/兑换码/队列状态等），不符合最小暴露原则。
2. 补录与人工介入缺少强SLA字段与明确处理时限。

## 3. P0整改任务（上线阻断项）

### P0-1 公开补录强鉴权

1. 新增“邮箱验证码（OTP）+ 短时会话票据”机制。
2. 候选查询与提交补录都要求合法票据，票据绑定 `email + ip + ua + ttl`。
3. 未通过鉴权统一返回模糊错误，不暴露订单存在性细节。

验收标准：

1. 无OTP/票据时无法查询候选或执行补录。
2. 票据过期、跨IP、跨邮箱重放均失败。
3. 错误信息不区分“邮箱不存在/订单不存在/状态不允许”。

### P0-2 补录幂等与次数控制

1. 引入“订单补录互斥锁”（按 `order_id` 锁定）。
2. 订单新增补录次数与最近补录时间字段，设置窗口内最大次数。
3. 同一订单处于处理中时，后续请求直接拒绝或复用处理中结果。

验收标准：

1. 并发触发同一订单补录，仅允许一次执行。
2. 频繁补录触发阈值后返回受控错误码。
3. 无重复消耗兑换码记录。

### P0-3 质保锚点固定化

1. 订单表新增 `warranty_anchor_at`（首次邀请/首次兑换时间）与 `warranty_service_days_snapshot`。
2. 质保判定统一基于锚点，不再使用可被重置的 `invite_sent_at/redeemed_at`。
3. 补录重置流程不得修改质保锚点。

验收标准：

1. 补录前后 `windowEndsAt` 保持一致（同一订单）。
2. 多次补录不会延长质保截止时间。
3. 历史订单迁移后，质保结果与业务规则一致。

### P0-4 邀请后异常分支纠正

1. 区分“邀请调用失败”和“邀请已提交但同步失败”两类错误。
2. 邀请已提交时不回退 `pending`，改为 `invited` 或新增 `invited_sync_pending`。
3. 增加后台可见的“待同步”标记与重试入口。

验收标准：

1. 邀请API返回成功后，订单状态不会回退到 `pending`。
2. 同步失败不会触发重复消耗兑换码。
3. 管理端可重试同步并完成收敛。

### P0-5 退回订单前撤销外部邀请

1. 退回动作前，若存在 `invited_account_id`，先尝试撤销该邮箱邀请。
2. 撤销失败时阻断退回并提示人工介入（或记录为 `manual_required`）。
3. 撤销成功后再执行库存释放和状态退回。

验收标准：

1. 退回后目标邮箱不应仍存在有效邀请。
2. 不出现库存已释放但邀请仍可接受的冲突状态。
3. 失败路径有可追踪审计记录。

### P0-6 公开接口限流与脱敏

1. `GET /orders/supplement/candidates` 接入与提交接口一致的限流。
2. 公开返回体去除 `alipayPassphrase`、`redemptionCode`、`queueState` 等敏感字段。
3. 错误码保留，错误文案改为统一安全文案。

验收标准：

1. 候选查询超频返回 `429` 且带 `Retry-After`。
2. 公开API响应不包含敏感字段。
3. 前端页面功能不受影响（仅显示必要字段）。

## 4. P1整改任务（稳定性与运营闭环）

1. 补录记录补充 SLA 字段：`manual_due_at`、`manual_owner`、`manual_status`。
2. 增加“补录安全看板”：失败率、超时单、重复触发拦截次数。
3. 管理端增加“补录链路追踪ID”，支持一键定位日志。
4. 增加灰度开关：可按渠道或比例启用新补录策略。

## 5. 数据与兼容迁移计划

## 5.1 数据结构变更（建议）

1. `alipay_redpack_orders`
   - `warranty_anchor_at DATETIME`
   - `warranty_service_days_snapshot INTEGER`
   - `supplement_attempt_count INTEGER DEFAULT 0`
   - `last_supplement_at DATETIME`
   - `supplement_lock_version INTEGER DEFAULT 0`
2. `alipay_redpack_supplements`
   - `trace_id TEXT`
   - `manual_due_at DATETIME`
   - `manual_owner TEXT`

### 5.2 回填策略

1. `warranty_anchor_at`：优先 `redeemed_at`，其次 `invite_sent_at`，最后 `created_at`。
2. `warranty_service_days_snapshot`：按当时订单规则回填，缺失则用默认配置。
3. 回填脚本需幂等，可重复执行。

## 6. 参数配置清单（上线前）

建议新增或确认以下配置：

1. `ALIPAY_REDPACK_PUBLIC_RATE_LIMIT_WINDOW_SEC`
2. `ALIPAY_REDPACK_PUBLIC_RATE_LIMIT_MAX`
3. `ALIPAY_REDPACK_SUPPLEMENT_REQUIRE_OTP=true`
4. `ALIPAY_REDPACK_SUPPLEMENT_OTP_TTL_SEC=300`
5. `ALIPAY_REDPACK_SUPPLEMENT_MAX_ATTEMPTS_PER_ORDER=3`
6. `ALIPAY_REDPACK_SUPPLEMENT_ATTEMPT_WINDOW_SEC=86400`
7. `ALIPAY_REDPACK_SUPPLEMENT_SYNC_RETRY_MAX=3`

## 7. 测试与验收计划

### 7.1 自动化测试（必须）

1. 公开补录鉴权测试（OTP、票据过期、重放、跨IP）。
2. 并发补录幂等测试（同订单并发100次）。
3. 质保锚点回归测试（补录不延长质保）。
4. 邀请成功但同步失败测试（状态不回退、无重复消耗）。
5. 退回撤销邀请测试（撤销失败阻断退回）。
6. 公开响应脱敏与限流测试。

### 7.2 人工回归（必须）

1. 公开页：邮箱查询候选 -> 选择订单 -> 补录成功/失败链路。
2. 后台页：处理订单、自动同步、退回订单、补录重试、人工关闭。
3. 库存核对：可用数、预占数、已消耗数与订单状态一致。

## 8. 发布与回滚策略

### 8.1 发布顺序

1. 先发数据库迁移（向后兼容）。
2. 再发后端接口（保留老字段兼容期）。
3. 最后发前端（适配脱敏返回与新鉴权）。

### 8.2 灰度策略

1. 先对 10% 流量启用 OTP 与新状态机。
2. 观察 24 小时无异常后全量。

### 8.3 回滚策略

1. 保留旧补录路径开关，异常时快速切回。
2. 新增字段不删除，仅停用读写逻辑。

## 9. Go/No-Go 上线检查清单

1. P0 六项全部完成并通过测试。
2. 公开接口已完成脱敏与统一错误语义。
3. 并发压测下无重复消耗记录。
4. 质保窗口在补录后不发生延长。
5. 退回链路已验证“先撤销邀请后释放库存”。
6. 监控与告警（429异常、补录失败率、人工积压）已接入。

3. 自动化测试用例与回归报告。
4. 运行手册与故障处置手册（含人工介入流程）。
