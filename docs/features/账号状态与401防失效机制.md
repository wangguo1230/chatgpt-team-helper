# 账号状态同步与 401 失效防护机制

本机制核心为了解决在运行过程中由于 OpenAI Token（Assess Token 及 Refresh Token）失效导致的前后台数据状态不一致、用户上车被阻断、以及管理后台显示滞后等问题。

## 1. 核心流程概述

当检测到 ChatGPT API 调用返回 `401 Unauthorized` 或 `403 Forbidden` 错误时，系统不仅不会简单抛出异常，还会尝试启动自动隔离与恢复机制：

1. **自动刷新**：如果账号配置了 `refresh_token`，系统会自动尝试去重新获取全新的 Token（通过 OAuth 换取）。如果刷新成功，新的 Token 将自动更新到数据库，原操作（如同步人数、拉取列表、上车操作）将会继续无感执行。
2. **并发控制与加固 (Double-Check)**：针对多用户同时遇到 401 触发并发刷新的情况，引入了内存级的 `withLocks` 互斥锁。只有第一个获得锁的请求会向 OpenAI 申请新 Token；当其余的并发请求获得锁后，会重新确认数据库中该账号的 Token 是否已被更新过，如果是，则跳过刷新直接重试拉取数据流程，极大地避免了网络拥堵及浪费。
3. **彻底失效隔离**：当以下任意一种情况发生时，系统判定 Token 已彻底失效（无法再修）：
   - 手动或自动刷新 OAuth Token 时 OpenAI 明确返回 400（例如 Refresh Token 已被吊销）。
   - 没有配置 `refresh_token`。
   - 自动刷新 Token 报错或遇到不可逆的网络问题导致刷新失败。
4. **状态持久化与管理后台显示**：一旦确定 Token 彻底失效，系统将强制执行两步同步处理：
   - 将账号从上车队列中强制下架（即修改数据库中对应的 `is_open = 0`）。
   - 将该账号的到期时间 (`expire_at`) 强制统一修改为 **1970 年 1 月 1 日**，以确保前端管理后台在进行数据渲染时，即使有缓存也能立刻识别此账号处于“过期（红色）”的失效状态。

## 2. 账号封号 (account_deactivated) 识别

由于 OpenAI API 有时通过 401 返回的 Body 明确提示 `account_deactivated`，此时系统不仅将账号下架，还会自动将其在系统中标记为封号（`is_banned = 1`，并设置 `is_open = 0`），以此向其他模块发出强烈信号，使得定期扫描清理和前台开放列表都能实时排除该问题账号。

## 3. 即时告警机制

传统的周期性清扫与状态检查（例如后台扫雷策略）因为有处理间隔（如 1 小时执行一次），通常对管理员的感知会有一定延迟。为了提升故障响应效率：

现在系统在上述核心异常（**严重封号 account_deactivated** 和 **Token 彻底失效**）发生时，将利用自带的 `sendAdminAlertEmail` 功能立刻向系统配置的 `ADMIN_ALERT_EMAIL` 邮箱发送一封即时告警邮件，包含：
- 失效的账号 ID 及邮箱
- 具体的触发原因（如 OAuth 刷新失败的错误细节等）
- 系统执行的操作说明（如已下架或已标记过期）

## 4. 其它注意事项

此机制已经覆盖到所有后台操作场景，包含：
- `/api/gpt-accounts/:id/refresh-token`（手动点击刷新）
- `/api/gpt-accounts/:id/sync-user-count`（手动同步人员数）
- `/api/gpt-accounts/check-status/stream`（手动批量测试连通性状态）
这些路由都被重构纳入到同一个底层的 Wrapper(`executeWithTokenRefresh`)，遵循统一的重试、失效与并发处理逻辑。

## 5. 兑换链路的懒校验与封号闭环

兑换链路新增了“按码懒校验”机制，目标是在不做全量账号扫描的前提下，确保当前使用的兑换码不会指向失效账号：

1. 在兑换码真正核销前，仅校验**当前选中兑换码所属账号**是否可访问。
2. 若检测到账号不可用（如 401/403/404 或 `account_deactivated` 等），系统会自动执行：
   - 账号标记封号并下架（`is_banned=1`、`is_open=0`）；
   - 删除该账号全部未使用兑换码；
   - 自动选码链路继续尝试下一个可用兑换码（懒重试），不做全量账号预检。
3. 告警由“即时多封”调整为“单次兑换流程汇总一封”，将以下内容聚合发送：
   - 本次封号账号列表；
   - 按渠道统计的低库存告警（阈值见下）。

相关环境变量：
- `REDEMPTION_LOW_STOCK_THRESHOLD`：兑换后按渠道低库存阈值（`0` 关闭）。
- `REDEEM_LAZY_RETRY_MAX_ATTEMPTS`：自动选码在账号不可用场景下的懒重试次数。
- `REDEMPTION_ALERT_EMAIL_SUBJECT`：兑换链路汇总告警邮件主题。
